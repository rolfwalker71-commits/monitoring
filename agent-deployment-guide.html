Monitoring Agent Guide

Deployment, Update und Payload-Referenz

Diese Seite ist in zwei getrennte Welten aufgebaut: zuerst Windows, danach Linux. Beide Bereiche folgen dem gleichen Ablauf: neue Agents deployen, einzelne Skripte beziehen, bestehende Agents updaten, Daten manuell senden, Lizenzpfade, Payload-Quellen und wichtige Defaults.

1. Windows

Installiert nach C:\ProgramData\monitoring-agent, mit Scheduled Tasks für Collect und Update sowie lokaler Queue im selben Basisverzeichnis.

2. Linux

Installiert nach /opt/monitoring-agent, Konfiguration unter /etc/monitoring-agent/agent.conf und Queue unter /var/lib/monitoring-agent/queue.

Wichtiger Unterschied

HANA-Abfragen existieren aktuell nur im Linux-Agent. Der Windows-Agent liest SAP-B1-/SQL-Server-Daten, aber keine HANA/hdbsql-Queries.

Ablauf pro Welt

1. Deploy

Ein neuer Host zieht das Install-Skript und erstellt damit die lokale Installation samt Konfiguration, Logs, Queue und Scheduler/Cron.

2. Einzelne Skripte

Jedes Kernskript kann direkt vom Server-Endpunkt /updates geholt werden, zum Beispiel für Reparatur, Debugging oder manuelle Verteilung.

3. Betrieb

Danach folgen Update, manuelles Senden, Lizenz- und Datenquellen. Beide Agents senden an /api/v1/agent-report und verwenden lokale Queue-Dateien bei Fehlern.

1. Windows

Neu deployen

$tmp = "$env:TEMP\install_agent.ps1"
(New-Object System.Net.WebClient).DownloadFile(
  'https://monitoring.example.com/updates/client/windows/install_agent.ps1',
  $tmp
)

powershell.exe -NoProfile -ExecutionPolicy Bypass -File $tmp `
  -ServerUrl 'https://monitoring.example.com' `
  -ApiKey 'MEIN_KEY' `
  -AgentId 'HOST01' `
  -DisplayName 'Produktiv Host' `
  -IntervalMinutes 15 `
  -UpdateHours 6
Installationspfad
C:\ProgramData\monitoring-agent
Config
C:\ProgramData\monitoring-agent\agent.conf
Queue
C:\ProgramData\monitoring-agent\queue
Tasks
monitoring-agent-collect, monitoring-agent-update

Install-Parameter

ParameterBedeutung
-ServerUrlPflichtparameter. Basis-URL des Monitoring-Servers.
-ApiKeyOptionaler API-Key.
-AgentIdExplizite Agent-ID.
-DisplayNameAnzeigename in der UI.
-IntervalMinutesCollect-Intervall, 1 bis 59.
-UpdateHoursUpdate-Intervall, 1 bis 24.
-RawBaseUrlExplizite Update-Basis, sonst ServerUrl/updates.

Einzelne Skripte direkt holen

$base = 'https://monitoring.example.com/updates/client/windows'
$target = 'C:\ProgramData\monitoring-agent'

(New-Object System.Net.WebClient).DownloadFile("$base/collect_and_send.ps1", "$target\collect_and_send.ps1")
(New-Object System.Net.WebClient).DownloadFile("$base/self_update.ps1", "$target\self_update.ps1")
(New-Object System.Net.WebClient).DownloadFile("$base/bootstrap_agent.ps1", "$env:TEMP\bootstrap_agent.ps1")
(New-Object System.Net.WebClient).DownloadFile("$base/install_agent.ps1", "$env:TEMP\install_agent.ps1")
(New-Object System.Net.WebClient).DownloadFile("$base/setup_harvest_sql_user.ps1", "$target\setup_harvest_sql_user.ps1")
(New-Object System.Net.WebClient).DownloadFile("$base/collect_and_scan_sap_tables.ps1", "$target\collect_and_scan_sap_tables.ps1")
(New-Object System.Net.WebClient).DownloadFile("$base/bulk_update_agents.ps1", "$env:TEMP\bulk_update_agents.ps1")
SkriptTypischer Zweck
install_agent.ps1Neuinstallation eines Windows-Agents.
bootstrap_agent.ps1Bestehende Installation reparieren oder frisch bootstrapen.
collect_and_send.ps1Metriken sammeln und Payload an den Server senden.
self_update.ps1In-Place-Update der Agent-Dateien.
setup_harvest_sql_user.ps1Harvest-SQL-User für SAP/SQL-Scan vorbereiten.
collect_and_scan_sap_tables.ps1Separater SAP-Tabellen-Scan.
bulk_update_agents.ps1Mehrere Windows-Agents aktualisieren.

Bestehende Agents updaten

powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\bootstrap_agent.ps1 `
  -ServerUrl https://monitoring.example.com `
  -DisableJitter

# Direktes In-Place-Update über self_update
powershell.exe -NoProfile -ExecutionPolicy Bypass -File \
  C:\ProgramData\monitoring-agent\self_update.ps1
  • bootstrap_agent.ps1 erkennt, ob bereits eine Installation existiert, und aktualisiert dann Skripte in place.
  • Wenn keine gültige Installation gefunden wird, zieht bootstrap_agent.ps1 das aktuelle install_agent.ps1 vom Server und installiert neu.

Daten manuell übermitteln

powershell.exe -NoProfile -ExecutionPolicy Bypass -File \
  C:\ProgramData\monitoring-agent\collect_and_send.ps1 -NoJitter

powershell.exe -NoProfile -ExecutionPolicy Bypass -File \
  C:\ProgramData\monitoring-agent\collect_and_send.ps1 -JitterMaxSec 30

powershell.exe -NoProfile -ExecutionPolicy Bypass -File \
  C:\ProgramData\monitoring-agent\collect_and_send.ps1 -DebugPayload
ParameterBedeutung
-NoJitterÜberspringt den Start-Delay.
-JitterMaxSecSetzt einen maximalen Jitter für diesen Lauf.
-DebugPayloadGibt den JSON-Payload auf stdout aus statt ihn zu senden.
Queue: Fehlgeschlagene Reports landen in C:\ProgramData\monitoring-agent\queue und werden später erneut gesendet.

Lizenzfile-Pfade

  • C:\ANG\Lizenzen\B01.txt
  • C:\ANG\Lizenz\B01.txt
  • C:\ANG\B01.txt

Der Windows-Agent nimmt den ersten vorhandenen Pfad und extrahiert daraus HARDWARE-KEY, INSTNO, EXPIRATION, SYSTEM-NR, CUSTOMER-NAME und CUSTOMER-NO.

Payload-Quellen

Der Windows-Agent füllt den SAP-B1-Teil aktuell über SQL Server und lokale SAP-Dateien, nicht über HANA / hdbsql.

# SAP B1 Version aus XML
C:\Program Files\SAP\SAP Business One DI API\Conf\InstallationConfigMSSQL.xml

# Extensions
SELECT AddOnName, Version
FROM [SLDModel.SLDData].[dbo].[Extensions]
ORDER BY AddOnName;

# SARI AddOns
SELECT AName, AddOnVer
FROM [SBO-COMMON].[dbo].[SARI]
ORDER BY AName;

# Fallback-Datenbankname
[SBOCOMMON].[dbo].[SARI]
Praktisch: Wenn du für Windows HANA dokumentieren willst, muss das erst implementiert werden. Die bestehenden Windows-Skripte greifen aktuell nur auf SQL-Server-/SAP-B1-Quellen zu.

2. Linux

Neu deployen

curl -fsSL https://monitoring.example.com/updates/client/linux/install_agent.sh \
  | sudo bash -s -- \
    --server-url https://monitoring.example.com \
    --api-key MEIN_KEY \
    --agent-id host.example.local \
    --display-name "Produktiv Host" \
    --interval-minutes 15 \
    --update-hours 6
Installationspfad
/opt/monitoring-agent
Config
/etc/monitoring-agent/agent.conf
Queue
/var/lib/monitoring-agent/queue
Logs
/var/log/monitoring-agent.log, /var/log/monitoring-agent-update.log

Install-Parameter

ParameterBedeutung
--server-urlPflichtparameter. Basis-URL des Monitoring-Servers.
--api-keyOptionaler API-Key für die REST-Calls.
--agent-idExplizite Agent-ID, sonst Hostname/FQDN.
--display-nameAnzeigename in der UI.
--interval-minutesCollect-Intervall, erlaubt 1 bis 59.
--update-hoursUpdate-Intervall, erlaubt 1 bis 24.
--collect-script-urlÜberschreibt die URL für collect_and_send.sh.
--raw-base-urlExplizite Update-Basis, sonst SERVER_URL/updates.
--insecure-tlsAktiviert curl --insecure; nur für kaputte CA-Ketten.

Einzelne Skripte direkt holen

base="https://monitoring.example.com/updates/client/linux"

curl -fsSL "$base/install_agent.sh" -o /tmp/install_agent.sh
curl -fsSL "$base/collect_and_send.sh" -o /tmp/collect_and_send.sh
curl -fsSL "$base/self_update.sh" -o /tmp/self_update.sh
curl -fsSL "$base/repair_agent_conf.sh" -o /tmp/repair_agent_conf.sh

chmod +x /tmp/install_agent.sh /tmp/collect_and_send.sh /tmp/self_update.sh /tmp/repair_agent_conf.sh
SkriptTypischer Zweck
install_agent.shNeuinstallation eines Linux-Agents.
collect_and_send.shMetriken sammeln und Payload an den Server senden.
self_update.shIn-Place-Update der Agent-Dateien.
repair_agent_conf.shFehlende Config-Werte und HANA-Defaults nachziehen.

Bestehende Agents updaten

sudo CONFIG_FILE=/etc/monitoring-agent/agent.conf \
  AGENT_VERSION_FILE=/opt/monitoring-agent/AGENT_VERSION \
  /opt/monitoring-agent/self_update.sh

sudo SERVER_URL=https://monitoring.example.com \
  /opt/monitoring-agent/repair_agent_conf.sh
  • self_update.sh lädt collect_and_send.sh, self_update.sh und AGENT_VERSION erneut vom Server.
  • repair_agent_conf.sh setzt fehlende Standardwerte wie HANA_SID, HANA_ADDONS_USER und Ports nach.

Daten manuell übermitteln

sudo /opt/monitoring-agent/collect_and_send.sh --no-jitter

sudo /opt/monitoring-agent/collect_and_send.sh --jitter-max-sec 30

sudo /opt/monitoring-agent/collect_and_send.sh --help
ParameterBedeutung
--no-jitterÜberspringt den deterministischen Start-Delay.
--jitter-max-sec NSetzt den maximalen Jitter für genau diesen Lauf.
Hinweis: Linux hat aktuell keinen eingebauten --print-payload-Parameter. Bei Sendeproblemen landet der JSON-Report in der Queue unter /var/lib/monitoring-agent/queue.

Lizenzfile-Pfade

Payload-Quellen

Version / Branch: Der Linux-Agent ruft HDB version auf, bevorzugt als <sid>adm. Die SID wird bei Bedarf aus /hana/shared/<SID> auto-detektiert.
# HANA-Version
HDB version

# HANA AddOns (Lightweight)
SELECT "NAME", "Version" FROM "SLDDATA"."EXTENSIONS";

# HANA AddOns (Legacy)
SELECT "AName", "AddOnVer" FROM "SBOCOMMON"."SARI";

# HANA DB Memory pro Schema
SELECT SCHEMA_NAME,
       ROUND(SUM(MEMORY_SIZE_IN_TOTAL) / 1024 / 1024 / 1024, 2) AS MEMORY_GB
FROM M_CS_TABLES
WHERE UPPER(SCHEMA_NAME) NOT LIKE 'SAP%'
  AND UPPER(SCHEMA_NAME) NOT IN ('SLDDATA','SBOCOMMON','LANDSCAPE')
  AND SCHEMA_NAME NOT LIKE '\_%' ESCAPE '\'
GROUP BY SCHEMA_NAME
HAVING SUM(MEMORY_SIZE_IN_TOTAL) > 0
ORDER BY MEMORY_GB DESC;
  • Transport über hdbsql als <sid>adm.
  • Default-Credentials: HARVEST / konfigurierbares Passwort in agent.conf.
  • Default-Host/Port: 127.0.0.1:30015, mit automatischer 3xx15-Port-Erkennung und lokalen Probe-Fallbacks.

Ziel-Endpoints

Report Upload
POST /api/v1/agent-report
Command Polling
GET /api/v1/agent-commands
Command Result
POST /api/v1/agent-command-result
  • Falls konfiguriert, wird der API-Key als Header X-Api-Key mitgesendet.
  • Wenn das Senden fehlschlägt, wird der Payload lokal als JSON gepuffert und beim nächsten Lauf erneut versucht.
