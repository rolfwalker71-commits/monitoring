Externe Monitore – Konzeptskizze

Empfehlung: kein Dummy-Host, sondern eigene Entität external_monitors mit serverseitigem Poller (Uptime-Kuma-ähnlich). Bestehende Agent-Hosts bleiben unverändert; externe Services erscheinen als eigener Block in der Sidebar und optional verknüpft im Host-Detail.

2. Datenmodell & API (MVP)

Erste Iteration: HTTP + SSL-Zertifikat, Admin-CRUD, Status-Liste, Alerts.

CREATE TABLE external_monitors ( id INTEGER PRIMARY KEY, name TEXT NOT NULL, monitor_type TEXT NOT NULL, -- http | tcp | ssl_cert target_url TEXT NOT NULL, customer_id INTEGER, related_host_uid TEXT, -- optional Link zum Agent-Host interval_sec INTEGER DEFAULT 300, expected_status INTEGER, -- z.B. 200 keyword TEXT, -- optional Inhalt-Check timeout_sec INTEGER DEFAULT 15, enabled INTEGER DEFAULT 1, created_at_utc TEXT, updated_at_utc TEXT ); CREATE TABLE external_monitor_results ( id INTEGER PRIMARY KEY, monitor_id INTEGER NOT NULL, checked_at_utc TEXT NOT NULL, status TEXT NOT NULL, -- up | down | degraded response_ms INTEGER, http_status INTEGER, cert_expires_at_utc TEXT, cert_days_left INTEGER, error_message TEXT );
  • GET /api/v1/external-monitors — Liste inkl. letztem Status
  • POST /api/v1/external-monitors — Monitor anlegen (Admin)
  • PUT /api/v1/external-monitors/:id — bearbeiten
  • GET /api/v1/external-monitors/:id/history — Sparkline / 24h
  • GET /api/v1/external-monitors/summary — KPI: up/down/cert-warn
  • — Worker-Thread in receiver.py (analog Agent-Ingest)
  • — Alert bei 2× fail oder cert_days_left < 14
SAP-typische Checks im MVP: Service Layer /b1s/v1/$metadata, Web Client-URL, TCP 50000, Zertifikat des Reverse Proxy.

4. Umsetzung in Phasen

PhaseInhaltAufwand (grob)
MVP 1 DB + Worker (HTTP/SSL) + Admin-Formular + Sidebar-Block „Services“ ~2–3 Tage
MVP 2 Alerts (Down/Zertifikat), KPI im Header, Verknüpfung related_host_uid, Tab im Host-Detail ~2 Tage
MVP 3 TCP-Port-Checks, Keyword, Verfügbarkeits-Historie, Mobile-Ansicht ~2–3 Tage
Warum kein Dummy-Host? Aktiv/Inaktiv-KPIs, Agent-Version-Balken, Filesystem-Alerts und SAP-Harvest würden verfälscht oder leer bleiben. Getrennte Entität + optionaler Link zum echten Host ist sauberer und wartbarer.
