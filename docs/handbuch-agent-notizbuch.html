1 · Windows-Agent installieren

PowerShell als Administrator öffnen. Der Agent landet unter C:\ProgramData\monitoring-agent und meldet sich per geplantem Task alle 15 Minuten (Standard).

Neuinstallation

$tmp = "$env:TEMP\install_agent.ps1"
Invoke-WebRequest -Uri 'https://infoboard.ang-schweiz.ch/updates/client/windows/install_agent.ps1' -UseBasicParsing -OutFile $tmp

powershell.exe -NoProfile -ExecutionPolicy Bypass -File $tmp `
  -ServerUrl 'https://infoboard.ang-schweiz.ch' `
  -ApiKey 'IHR_API_KEY' `
  -AgentId 'HOST01.domain.local' `
  -DisplayName 'Kundenname Server' `
  -IntervalMinutes 15 `
  -UpdateHours 1

Falls Invoke-WebRequest wegen TLS/Revocation scheitert, curl.exe -fsSL --ssl-no-revoke … nutzen (siehe README im Repo).

Wichtige Pfade

WasWo
KonfigurationC:\ProgramData\monitoring-agent\agent.conf
Collect-Skriptcollect_and_send.ps1
Collect-Taskmonitoring-agent-collect (läuft als SYSTEM)
Update-Taskmonitoring-agent-update
Logmonitoring-agent.log

Test nach der Installation

powershell.exe -NoProfile -ExecutionPolicy Bypass -File `
  C:\ProgramData\monitoring-agent\collect_and_send.ps1 -NoJitter

Im Log: HTTP 202 oder Payload queued. Danach Host-Karte im Infoboard prüfen.

Reparatur / Skripte nachziehen

powershell.exe -NoProfile -ExecutionPolicy Bypass -File `
  C:\ProgramData\monitoring-agent\bootstrap_agent.ps1 `
  -ServerUrl 'https://infoboard.ang-schweiz.ch'

2 · Linux-Agent installieren

Installation per curl als root (sudo). Skripte unter /opt/monitoring-agent, Konfiguration unter /etc/monitoring-agent/agent.conf.

Neuinstallation

curl -fsSL 'https://infoboard.ang-schweiz.ch/updates/client/linux/install_agent.sh' | sudo bash -s -- \
  --server-url 'https://infoboard.ang-schweiz.ch' \
  --api-key 'IHR_API_KEY' \
  --agent-id 'host.domain.local' \
  --display-name 'Kundenname Server' \
  --interval-minutes 15 \
  --update-hours 1

Wichtige Pfade

WasWo
Skripte/opt/monitoring-agent/
Konfiguration/etc/monitoring-agent/agent.conf
Cron/etc/cron.d/monitoring-agent
Log/var/log/monitoring-agent.log
Queue/var/lib/monitoring-agent/queue/

Test nach der Installation

sudo /opt/monitoring-agent/collect_and_send.sh --no-jitter
sudo tail -n 30 /var/log/monitoring-agent.log
Wichtig: HANA-Abfragen (hdbsql) gibt es nur auf Linux.
Windows liefert SAP B1 / SQL Server – kein HANA-Memory über den Windows-Agent.

3 · Harvest-User unter Windows (SQL Server)

Für SAP Business One auf SQL Server braucht der Agent einen SQL-Login harvest. Damit werden u. a. gelesen:

  • Datenbankgrößen und Extensions (über Harvest-Verbindung)
  • SARI-AddOns und SAP-Tabellen-Scan (ENABLE_SAP_SCAN=1)

Automatisches Setup (empfohlen)

Als Administrator auf dem SQL-Host:

powershell.exe -NoProfile -ExecutionPolicy Bypass -File `
  C:\ProgramData\monitoring-agent\setup_harvest_sql_user.ps1

Das Skript versucht:

  1. SQL-Instanz zu finden (localhost, Named Instance, …)
  2. Login harvest anzulegen
  3. Server-Rechte (VIEW SERVER STATE, …) und db_datareader auf User-DBs zu vergeben
  4. HARVEST_SQL_* und ENABLE_SAP_SCAN=1 in agent.conf zu schreiben

agent.conf – relevante Keys

KeyBeispiel / Hinweis
HARVEST_SQL_SERVERlocalhost oder .\SQLEXPRESS
HARVEST_SQL_USERharvest
HARVEST_SQL_PASSWORDProjektpasswort (Standard im Setup-Skript)
HARVEST_SETUP_SQL_ADMIN_USEROptional: sa-ähnlicher Admin, falls Windows-Auth für Setup nicht reicht
HARVEST_SETUP_SQL_ADMIN_PASSWORDPasswort zum Admin-Login
ENABLE_SAP_SCAN1 für Tabellen-Scan

Manuell prüfen

powershell.exe -NoProfile -ExecutionPolicy Bypass -File `
  C:\ProgramData\monitoring-agent\collect_and_send.ps1 -NoJitter

Im Report-JSON: Abschnitt harvest_status – dort stehen can_connect, databases_accessible und Fehlertexte.

4 · Harvest-User unter Linux (HANA)

Auf Linux heisst der Harvest-User HARVEST – ein HANA-Datenbankbenutzer (nicht SQL Server). Der Agent liest damit AddOns und die Tenant-Datenbankliste per hdbsql.

SQL-Skript (mitgeliefert)

Pfad auf dem HANA-Host nach Agent-Installation:

/opt/monitoring-agent/setup_harvest_hana_user.sql

Das Skript legt User HARVEST an und vergibt SELECT auf die Schemas SLDDATA und SBOCOMMON (Tabellen Extensions, SARI, COMPANYDBS).

In SAP HANA Studio einrichten

  1. HANA Studio (oder Database Explorer) öffnen und System hinzufügen.
  2. Als Admin verbinden (z. B. SYSTEM oder gleichberechtigter User).
  3. Zum Tenant wechseln, in dem die B1-Daten liegen – nicht SYSTEMDB.
    Bei Multitenant: pro Tenant wiederholen (z. B. BUP, PRD).
  4. SQL Console öffnen (Rechtsklick auf Tenant → Open SQL Console).
  5. Skript laden: File → Open → /opt/monitoring-agent/setup_harvest_hana_user.sql
  6. Passwort im Skript prüfen – muss mit HANA_ADDONS_PASSWORD in agent.conf übereinstimmen (Standard wie bei Windows-Harvest).
  7. Skript ausführen (F8 / Execute).
  8. Optional verifizieren (in neuer Console als HARVEST):
    SELECT COUNT(*) FROM "SLDDATA"."COMPANYDBS";
Multitenant: User und Grants in jedem Tenant setzen, aus dem der Agent Daten lesen soll.
Der Agent erkennt Tenants automatisch unter /usr/sap/<SID>/SYS/global/hdb/custom/config/tenants/.

Alternative: Skript per hdbsql (ohne Studio)

# Beispiel SID BUP, Tenant BUP, als <sid>adm und SQL-Admin:
su - bupadm -c "hdbsql -d BUP -u SYSTEM -p 'ADMIN_PASS' \
  -I /opt/monitoring-agent/setup_harvest_hana_user.sql"

# Verbindung als HARVEST testen:
su - bupadm -c "hdbsql -d BUP -u HARVEST -p 'PASS' \
  \"SELECT COUNT(*) FROM \\\"SLDDATA\\\".\\\"COMPANYDBS\\\"\""

agent.conf – HANA-Keys

KeyBedeutung
HANA_SIDSystem-ID, z. B. BUP (ls /hana/shared)
HANA_ADDONS_USERStandard: HARVEST
HANA_ADDONS_PASSWORDPasswort aus dem SQL-Skript
HANA_ADDONS_HOSTMeist 127.0.0.1
HANA_ADDONS_PORTSQL-Port, z. B. 30015 (3<Inst>15)
HANA_ADDONS_ENABLED1 (Standard bei Install)

Nach dem Setup prüfen

  1. HANA_SID und Passwort in /etc/monitoring-agent/agent.conf setzen (falls abweichend)
  2. sudo /opt/monitoring-agent/collect_and_send.sh --no-jitter
  3. Im Infoboard: HANA-AddOns und Datenbankliste im Report sichtbar?
Symptom HANA leer, Rest OK?
→ Falscher Tenant, SID/Port, oder HARVEST ohne Grants auf SLDDATA/SBOCOMMON.
Log: /var/log/monitoring-agent.log

5 · SQL-Datenbankgrößen fehlen im Infoboard

Gilt für Windows / SQL Server. Typisches Bild: Instanz und Version sind sichtbar, aber die Tabelle mit DB-Größen ist leer.

Warum zwei Kontexte?

KontextWofür
harvest (SQL-Login)SAP B1 Harvest-Daten, Extensions, Tabellen-Scan
NT AUTHORITY\SYSTEMGrößenabfrage über sys.master_files – der Collect-Task läuft als SYSTEM!

Harvest allein reicht für die Größenanzeige oft nicht. SYSTEM braucht Server-Rechte.

Symptome

  • Einzelmeldung → Datenbanken: keine MB/GB-Zeilen
  • Diagnose zeigt z. B. SYSTEM_USER: NT AUTHORITY\SYSTEM (DE: NT-AUTORITÄT\SYSTEM)
  • Abfrage auf sys.master_files liefert nichts oder Permission denied

Schritt 1 – Diagnose im Infoboard

Host öffnen → Report → Karte Datenbanken aufklappen → Bereich SQL Authentifizierung (für Diag). Dort notieren:

  • SYSTEM_USER
  • ORIGINAL_LOGIN
  • SUSER_SNAME

Das UI schlägt ein Grant-Skript vor – Werte exakt aus der Diagnose übernehmen!

Schritt 2 – GRANT in SQL Server (als Sysadmin)

-- Login anlegen (Name aus SYSTEM_USER – ggf. NT-AUTORITÄT\SYSTEM)
IF SUSER_ID(N'NT-AUTORITÄT\SYSTEM') IS NULL
    CREATE LOGIN [NT-AUTORITÄT\SYSTEM] FROM WINDOWS;

-- Falls ORIGINAL_LOGIN ein Maschinenkonto ist (z. B. DOMÄNE\HOST$):
-- IF SUSER_ID(N'DOMÄNE\HOST$') IS NULL
--     CREATE LOGIN [DOMÄNE\HOST$] FROM WINDOWS;

GRANT VIEW SERVER STATE TO [NT-AUTORITÄT\SYSTEM];
GRANT VIEW ANY DEFINITION TO [NT-AUTORITÄT\SYSTEM];

-- Optional für Maschinenkonto:
-- GRANT VIEW SERVER STATE TO [DOMÄNE\HOST$];
-- GRANT VIEW ANY DEFINITION TO [DOMÄNE\HOST$];

Schritt 3 – Verifizieren

  1. collect_and_send.ps1 -NoJitter ausführen
  2. Neue Meldung im Infoboard: DB-Tabelle mit Größen

Wenn es danach noch leer ist

  • setup_harvest_sql_user.ps1 erneut – Harvest-Verbindung prüfen
  • HARVEST_SQL_SERVER: richtige Instanz? (.\SQLEXPRESS, Named Instance, SQL Browser)
  • Firewall / TCP vs. Named Pipes
  • Collect-Log: C:\ProgramData\monitoring-agent\monitoring-agent.log

6 · Abschluss-Checkliste

  • Agent installiert (Windows oder Linux)
  • Manueller Collect mit -NoJitter / --no-jitter → HTTP 202
  • Host erscheint im Infoboard, letzte Meldung plausibel
  • Windows: setup_harvest_sql_user.ps1 gelaufen, harvest_status.can_connect=true
  • Windows + SQL: GRANT an SYSTEM (Kap. 5), DB-Größen sichtbar
  • Linux + HANA: setup_harvest_hana_user.sql im Tenant ausführen, HANA_SID + Passwort in agent.conf
  • Geplanter Task / Cron aktiv (nicht nur manueller Test)
