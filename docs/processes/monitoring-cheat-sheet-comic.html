Die Kurzfassung für den Alltag: Welche Skripte sind wichtig, welche Parameter musst du kennen und wie läuft der Pfad von Agent bis Alarm wirklich durch das System.

🎯 Ziel

Schnell entscheiden

Was deploye ich, welches Skript starte ich und an welcher Stelle im Ablauf hängt ein Problem?

🧰 Kernobjekte

Agent, Receiver, DB, Alerts

Linux und Windows Agents liefern Reports an den Server; dort folgen Persistenz, Alerting, Mail und Dashboard.

⚡ Merke

Install → Collect → Queue → Ingest → Alert → Update

Das ist der Gesamtpfad, den du für Debugging fast immer von links nach rechts prüfen kannst.

1. Der Gesamtfluss in 6 Panels

BAM! End-to-End
01

Deploy

Windows: install_agent.ps1 oder bootstrap_agent.ps1

Linux: install_agent.sh

Konfig, Pfade, Scheduler/Cron und Update-Routen werden eingerichtet.

02

Collect

collect_and_send.ps1 bzw. collect_and_send.sh sammeln Host-, FS-, DB- und SAP-Daten.

03

Queue bei Fehlern

Wenn der Server nicht erreichbar ist, landen Payloads lokal in der Agent-Queue und werden später nachgesendet.

04

Report Ingest

/api/v1/agent-report validiert JSON, prüft Auth/Grace-Mode und gibt in die DB-Pipeline weiter.

05

Persistenz & Alerting

Reports, Filesystems, Lifecycle und offene Alerts werden aktualisiert; Mail/Telegram kann folgen.

06

Self-Update

self_update.ps1 und self_update.sh laden neue Skripte von /updates.

2. Windows Agent Cheats

🪟 Scheduled Tasks
Pfad: C:\ProgramData\monitoring-agent Tasks: monitoring-agent-collect, monitoring-agent-update
  • install_agent.ps1: Neuinstallation mit Config, Queue und Scheduled Tasks.
  • bootstrap_agent.ps1: Bestehende Installation reparieren oder notfalls neu ziehen.
  • collect_and_send.ps1: Daten sammeln und an den Server senden.
  • self_update.ps1: Agent-Dateien in place aktualisieren.
  • setup_harvest_sql_user.ps1: SQL-Harvest-User vorbereiten.
  • bulk_update_agents.ps1: Mehrere Windows Agents aktualisieren.
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\install_agent.ps1 
  -ServerUrl https://monitoring.example.com 
  -ApiKey MEIN_KEY 
  -IntervalMinutes 15 
  -UpdateHours 6

3. Linux Agent Cheats

🐧 Cron
Pfad: /opt/monitoring-agent Config: /etc/monitoring-agent/agent.conf
  • install_agent.sh: Erstinstallation inkl. Cron, Config und Download der Kernskripte.
  • collect_and_send.sh: Metriken sammeln und Payload senden.
  • self_update.sh: Lädt neue Skripte und Agent-Version nach.
  • repair_agent_conf.sh: Fehlende Defaults und HANA-Werte nachziehen.
curl -fsSL https://monitoring.example.com/updates/client/linux/install_agent.sh \
  | sudo bash -s -- \
    --server-url https://monitoring.example.com \
    --api-key MEIN_KEY \
    --interval-minutes 15 \
    --update-hours 6

4. Wichtigste Parameter

🧪 Nur das Relevante

Gemeinsame Pflicht

ServerUrl bzw. --server-url

Basis-URL des Monitoring-Servers. Ohne diese Adresse gibt es keinen Deploy und keinen Report-Upload.

Auth

ApiKey bzw. --api-key

Optional, aber für abgesicherte Agent-Calls praktisch der Regelfall.

Taktung

IntervalMinutes bzw. --interval-minutes

Sammler-Rhythmus, typischerweise 15 Minuten, gültig 1 bis 59.

Update

UpdateHours bzw. --update-hours

Update-Intervall der Agents, gültig 1 bis 24.

Identity

AgentId, DisplayName

Steuern Host-Identität und Anzeigenamen im Monitoring.

Sonderwege

RawBaseUrl, --raw-base-url, --collect-script-url, --insecure-tls

Für Proxy-/Bootstrap-Sonderfälle oder kaputte Zertifikatsketten im Notfall.

5. Wo prüfe ich was?

🔎 Debug-Route
Deploy-Problem: Install-Skript, lokale Pfade, Task/Cron-Eintrag, Config-Datei.
Keine Daten: Collector manuell starten, Logs prüfen, Queue-Verzeichnis prüfen.
Server nimmt nichts an: /api/v1/agent-report, API-Key, Grace-Mode, Pflichtfelder.
Kein Alarm: Thresholds, Mute/Ack/Blacklist, Routing-Matrix, Mail/OAuth.
Update hängt: /updates-Pfad, lokale Version, Update-Task/Cron, Self-Update-Logs.
