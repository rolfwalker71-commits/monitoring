2. Was steuert den Alarm?

🧭 Alerting
  • Severity: Warning, Critical oder Escalated bestimmen Priorität.
  • Mute/Ack: Unterdrücken Versand oder markieren aktive Bearbeitung.
  • Blacklist/Visibility: Kann einzelne Filesysteme komplett aus der Kette nehmen.
  • Resolution: Normalisierte Werte schließen offene Alerts automatisch.
Wichtige Stellen im Code:
- update_alerts_for_report
- evaluate_severity_for_thresholds
- maybe_send_alert_message
- maybe_send_alert_reminders

3. Was steuert den Versand?

📨 Mail Routing
Instant Alert

Sofort nach Alert-Event, wenn Kanal und Mindest-Schweregrad passen.

Reminder

Periodisch für weiterhin offene Alerts.

Digest

Gebündelte Übersicht aus dem Scheduler, last_sent nur bei Erfolg.

Customer Alert

Spezialfall mit dediziertem Trigger und Test-Endpunkt.

4. OAuth Kurzlogik

🔐 Token Leben
  • Connect: User verbindet Microsoft-Konto über den OAuth Flow.
  • Callback: Server speichert Refresh-/Access-Daten.
  • Refresh: Vor Versand wird ein frischer Access Token sichergestellt.
  • Disconnect: Verbindung kann wieder entfernt werden; Versand stoppt dann sauber.
Wichtige Funktionen:
- ensure_microsoft_access_token
- send_microsoft_mail
- send_microsoft_mail_multi
- maybe_send_scheduled_user_mails

5. Fehlerbild → erste Prüfung

🛠 Debug

Alert fehlt

Thresholds, Blacklist und Alert-Update-Pfad prüfen.

Mail fehlt

Routing-Matrix, Severity-Filter und Empfängereinstellungen prüfen.

OAuth defekt

Token-Refresh, Connect-Status und Callback-Daten prüfen.

Reminder fehlt

Open-Status, Scheduler-Intervall und letzte Versandmarke prüfen.

Vertiefung dazu in mailversand-prozess.html, alert-handling-gesamtprozess.html, oauth-lifecycle.html und alert-routing-matrix.html.
