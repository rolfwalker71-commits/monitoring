SYSTEM Infoboard – Betrieb & Agent-Rollout

Dieses Handbuch richtet sich an neue Kolleginnen und Kollegen im technischen Betrieb. Ziel: Linux- und Windows-Agents ausrollen, HANA- und SQL-Harvest konfigurieren, typische Infoboard-Funktionen nutzen und bekannte Stolpersteine (z. B. fehlende SQL-Datenbankgrößen) selbst beheben.

Screenshots: Die Abbildungen unten stammen aus der lokalen Instanz (http://127.0.0.1:8080) bzw. aus UI-Mockups mit echten Styles. Für Schulungen in Produktion können die PNG-Dateien unter docs/assets/ durch Screenshots aus eurer Umgebung ersetzt werden.

Du kannst danach …

  • Windows- und Linux-Agents installieren und updaten
  • agent.conf für HANA/SQL anpassen
  • SQL-Grants für NT AUTHORITY\SYSTEM setzen
  • Hosts, Alerts und Mobile-App bedienen

Wichtig

HANA-Abfragen (hdbsql) gibt es nur auf Linux. Windows liefert SAP B1 / SQL Server, nicht HANA-Memory-Schemas.

Pfade

  • Infoboard-URL: z. B. https://infoboard.ang-schweiz.ch
  • Updates: {SERVER_URL}/updates
  • API-Upload: POST /api/v1/agent-report

3. Infoboard – Standardfunktionen

Die Weboberfläche ist in Übersicht (Host-Karten, Filter, KPIs) und Globale Ansichten (Alerts, Trends, inaktive Hosts, …) gegliedert. Rechtsklick auf Host-Karten öffnet Kontextaktionen (Update, Lizenz, Ein-/Ausblenden).

BereichFunktionTypischer Use-Case
Host-ÜbersichtKarten mit Kunde, Status-Punkt, SAP/HANA-ChipsSchneller Gesundheitscheck aller Kunden
Host-FilterSuche, OS, Land, Sortierung, Interessen„Alle Prod-Server in DE, zuletzt gemeldet“
KPI-LeisteOffen, Kritisch, Aktiv, Inaktiv, DB-KPIsLagebild ohne einzelne Hosts zu öffnen

4. Windows-Agent ausrollen

Neuinstallation (empfohlen)

$tmp = "$env:TEMP\install_agent.ps1"
Invoke-WebRequest -Uri 'https://infoboard.ang-schweiz.ch/updates/client/windows/install_agent.ps1' -UseBasicParsing -OutFile $tmp

powershell.exe -NoProfile -ExecutionPolicy Bypass -File $tmp `
  -ServerUrl 'https://infoboard.ang-schweiz.ch' `
  -ApiKey 'IHR_API_KEY' `
  -AgentId 'HOST01.domain.local' `
  -DisplayName 'Kundenname Server' `
  -IntervalMinutes 15 `
  -UpdateHours 1
KomponentePfad / Name
InstallationC:\ProgramData\monitoring-agent
KonfigurationC:\ProgramData\monitoring-agent\agent.conf
Collect-Taskmonitoring-agent-collect (SYSTEM, alle 15 min)
Update-Taskmonitoring-agent-update
Log CollectC:\ProgramData\monitoring-agent\monitoring-agent.log
QueueC:\ProgramData\monitoring-agent\queue\

Manueller Test

powershell.exe -NoProfile -ExecutionPolicy Bypass -File `
  C:\ProgramData\monitoring-agent\collect_and_send.ps1 -NoJitter

# Danach self_update testen:
powershell.exe -NoProfile -ExecutionPolicy Bypass -File `
  C:\ProgramData\monitoring-agent\self_update.ps1

Interaktive PowerShell überspringt Jitter automatisch. Der geplante Task nutzt Jitter (Standard bis 300 s) – Task-Zeitlimit im Aufgabenplaner prüfen (mind. 15 min empfohlen).

Bootstrap / Reparatur

powershell.exe -NoProfile -ExecutionPolicy Bypass -File `
  C:\ProgramData\monitoring-agent\bootstrap_agent.ps1 `
  -ServerUrl 'https://infoboard.ang-schweiz.ch'

Lädt fehlende Skripte von /updates nach. Collect-Task wird bei Install neu angelegt, nicht bei jedem self_update synchronisiert.

5. Linux-Agent ausrollen

curl -fsSL 'https://infoboard.ang-schweiz.ch/updates/client/linux/install_agent.sh' | sudo bash -s -- \
  --server-url 'https://infoboard.ang-schweiz.ch' \
  --api-key 'IHR_API_KEY' \
  --agent-id 'host.domain.local' \
  --display-name 'Kundenname Server' \
  --interval-minutes 15 \
  --update-hours 1
KomponentePfad
Skripte/opt/monitoring-agent/
Config/etc/monitoring-agent/agent.conf
Cron/etc/cron.d/monitoring-agent
Log/var/log/monitoring-agent.log
Queue/var/lib/monitoring-agent/queue/

Manueller Test

sudo /opt/monitoring-agent/collect_and_send.sh --no-jitter
sudo CONFIG_FILE=/etc/monitoring-agent/agent.conf \
  /opt/monitoring-agent/self_update.sh

Cron reparieren (ab Agent 1.7.381)

sudo CONFIG_FILE=/etc/monitoring-agent/agent.conf INSTALL_DIR=/opt/monitoring-agent \
  REPAIR_CRON_SCHEDULE_ONLY=1 /opt/monitoring-agent/self_update.sh

sudo cat /etc/cron.d/monitoring-agent

Erste Zeile muss SHELL=/bin/bash sein – keine Schedule-Zeile mit SHELL= vermischen.

6. HANA-Anpassungen (nur Linux)

HANA-Daten (Version, AddOns, Schema-Memory) werden per hdbsql als User <SID>adm bzw. konfiguriertem DB-User geholt.

Schlüssel in agent.confBedeutungDefault
HANA_SIDSAP HANA System-IDAuto aus /hana/shared/<SID>
HANA_ADDONS_ENABLEDAddOn-Queries1
HANA_ADDONS_USERSQL-User für hdbsqlHARVEST
HANA_ADDONS_PASSWORDPasswort(projektspezifisch)
HANA_ADDONS_HOSTSQL Endpoint127.0.0.1
HANA_ADDONS_PORTPort (3xx15)30015

Typisches Vorgehen

  1. SID prüfen: ls /hana/shared → z. B. BUP
  2. In /etc/monitoring-agent/agent.conf setzen: HANA_SID="BUP"
  3. DB-User HARVEST in HANA anlegen (SELECT auf SLDDATA/SBOCOMMON – siehe Agent-Queries in Deployment Guide)
  4. sudo /opt/monitoring-agent/repair_agent_conf.sh oder self_update.sh für fehlende Keys
  5. Collect mit --no-jitter, im Report HANA-Abschnitt prüfen
Symptom: HANA leer, Rest OK → meist SID, Port, Firewall localhost, oder Rechte des HARVEST-Users. Log: /var/log/monitoring-agent.log.

7. SQL Harvest User (Windows / SAP B1)

Für Datenbankgrößen, Extensions und SARI-AddOns nutzt der Windows-Agent einen SQL-Login harvest (oder Windows-Auth über SYSTEM – siehe nächstes Kapitel).

Setup-Skript ausführen

powershell.exe -NoProfile -ExecutionPolicy Bypass -File `
  C:\ProgramData\monitoring-agent\setup_harvest_sql_user.ps1

Legt Login an, vergibt Server- und DB-Rechte (best effort), schreibt HARVEST_SQL_* in agent.conf.

agent.confZweck
HARVEST_SQL_SERVERInstanz, z. B. localhost oder .\SQLEXPRESS
HARVEST_SQL_USER / HARVEST_SQL_PASSWORDSQL-Auth für Harvest-Queries
HARVEST_SETUP_SQL_ADMIN_USEROptional: sa-ähnlicher Admin für GRANTs
HARVEST_SETUP_SQL_ADMIN_PASSWORDPasswort zum Admin-Login
ENABLE_SAP_SCANSAP-Tabellen-Scan ein/aus

8. Fehlende Datenbankgrößen – Grant an NT AUTHORITY\SYSTEM

Häufiger SAP-B1-Fall

Symptom

  • Im Infoboard unter Einzelmeldung → Datenbanken erscheint SQL-Instanz/Version, aber keine Größen oder leere Liste
  • Im Report steht unter Diagnose z. B. SYSTEM_USER: NT AUTHORITY\SYSTEM
  • sys.master_files liefert 0 Zeilen oder Verbindungsfehler

Der Collect-Task läuft als SYSTEM. Für Größenabfragen braucht dieser Kontext Server-Rechte VIEW SERVER STATE und VIEW ANY DEFINITION (nicht nur der Harvest-SQL-User).

Diagnose im Infoboard

In der DB-Karte aufklappen: SQL Authentifizierung (für Diag) – dort stehen SYSTEM_USER, ORIGINAL_LOGIN, SUSER_SNAME. Das vorgeschlagene Skript im UI an diese Werte anpassen.

SQL-Skript (Vorlage – Werte anpassen!)

-- 1) Windows-Logins anlegen (falls noch nicht vorhanden)
IF SUSER_ID(N'NT AUTHORITY\SYSTEM') IS NULL
    CREATE LOGIN [NT AUTHORITY\SYSTEM] FROM WINDOWS;

-- Falls der Agent unter Maschinenkonto läuft (ORIGINAL_LOGIN aus Diagnose):
-- IF SUSER_ID(N'IHRDOMAIN\IHRHOST$') IS NULL
--     CREATE LOGIN [IHRDOMAIN\IHRHOST$] FROM WINDOWS;

-- 2) Server-Rechte für Größenabfragen (master_files)
GRANT VIEW SERVER STATE TO [NT AUTHORITY\SYSTEM];
GRANT VIEW ANY DEFINITION TO [NT AUTHORITY\SYSTEM];

-- Optional zweites Konto aus ORIGINAL_LOGIN:
-- GRANT VIEW SERVER STATE TO [IHRDOMAIN\IHRHOST$];
-- GRANT VIEW ANY DEFINITION TO [IHRDOMAIN\IHRHOST$];

Auf deutschen Windows-Installationen heißt der Login im UI oft NT-AUTORITÄT\SYSTEM – im SQL-SSMS exakt den Namen aus SYSTEM_USER verwenden.

Nach dem Grant

  1. Als sa / Sysadmin ausführen
  2. collect_and_send.ps1 -NoJitter
  3. Neue Meldung im Infoboard: DB-Tabelle mit Größen (MB/GB)

Wenn weiterhin leer

  • Harvest-User (harvest) separat mit setup_harvest_sql_user.ps1 prüfen
  • SQL Browser / Instanzname in HARVEST_SQL_SERVER
  • Firewall, Named Pipes vs. TCP
